00185: RELATIONAL DATABASE DESIGN SPECIFICATION
00186: STEP 1: Entity Class → Database Table
00187: At this stage, the system retains only the entity classes that store persistent data and eliminates intermediate classes such as Controller, Service, or Repository.
00188: Entity Class User → Table User: Stores login account information and basic user profiles for the system.
00189: Entity Class Role → Table Role: Stores a list of roles (Admin, Manager, Employee,...) used in the system.
00190: Entity Class UserRole → Table User_Role:Storing data that maps users to roles serves the purpose of user permission management.
00191: Entity Class RolePermission → Table Role_Permission:Store data that maps roles to the permissions granted to those roles.
00192: STEP 2: Primary Key (PK) Definition
00193: Each table is assigned a primary key (PK) to uniquely identify each record.
00194: User (user_id, username, password_hash, email, full_name, phone, status, created_at)
00195: Role (role_id, role_name, description)
00196: User_Role (user_role_id, user_id, role_id)
00197: Role_Permission (role_permission_id, role_id, permission_id)
00198: Engineering architecture:
00199: All xxx_id identifier fields are implemented using the Surrogate Key mechanism, employing the INT AUTO_INCREMENT (or IDENTITY depending on the database management system) data type to optimize query performance, indexing, and future system scalability.
00200: STEP 3: Foreign Key (FK) for Associations
00201: Based on the relationships in the Class Diagram and ERD, the foreign keys are set up as follows:
00202: The User → User_Role (1 – Many) relationship
00203: A user can be assigned multiple roles.
00204: Solution: Set the foreign key user_id in the User_Role table to reference the primary key user_id of the User table.
00205: Relationship Role → User_Role (1 – Many)
00206: A role can be assigned to multiple users.
00207: Solution: Set the foreign key role_id in the User_Role table to reference the primary key role_id of the Role table.
00208: Relationship Role → Role_Permission (1 – Many)
00209: A single role can possess various levels of authority.
00210: Solution: Set the foreign key role_id in the Role_Permission table to reference the primary key role_id of the Role table.
00211: STEP 4: Aggregation / Composition Hierarchy
00212: Systemic relationship:
00213: Role ◇ → Role_Permission
00214: Each Role acts as the overall entity (Whole), containing a set of permissions linked through the Role_Permission table.
00215: Structural solution: This relationship is implemented using the foreign key role_id in the Role_Permission table.
00216: This foreign key establishes a Referential Integrity Constraint with the Role table, ensuring that all assigned permissions belong to a valid role that exists in the system.
00217: STEP 5: Generalization / Specialization Hierarchy (Inheritance)
00218: The Admin Management system does not implement a data inheritance mechanism between entities.
00219: All user account information is stored directly in the User table, while user classification into Admin, Manager, or Employee is done through the Role-Based Access Control (RBAC) mechanism.
00220: This mechanism uses the User_Role table to associate users with roles instead of building separate inheritance tables.
00221: Evaluating the solution:
00222: RBAC design enables a simple, flexible, maintainable, and scalable database structure. When new roles arise in the future, the system only needs to add the data to the Role table without changing the structure of the User table.

00280: Pattern 11. Name:Facade Pattern (Structural Pattern)
00281: The Facade Pattern provides a unified interface for a set of classes in a subsystem. Instead of the Controller interacting with many different objects, the Controller only works with a single Facade.
00282: 2. ProblemIn function Manage Account When an administrator creates an account or resets a password, the system has to work with several components:

00284: Role

00289: If AccountController communicates directly with all the classes above, then:
00290: The controller depends on too many classes.
00291: Coupling between components increases.
00292: The system is difficult to maintain and expand.
00293: Every change in the account creation or password reset process affects the Controller.
00294: The problem is that we need a mechanism that allows the Controller to work with only a single object instead of having to manage the entire subsystem.

00297: The controller only calls high-level operations:

00303: Account

00310: Benefits

00622: 10.1. Facade Pattern

00669: Operation 1 – softDeleteAccount()
00670: FUNCTION softDeleteAccount(accountId, adminId, reason)    IF adminId does not have MANAGE_ACCOUNT permission THEN        THROW UnauthorizedOperationException    END IF    IF accountId equals adminId THEN        THROW SelfDeletionNotAllowedException    END IF    IF reason is null OR reason is empty THEN        THROW DeletionReasonRequiredException    END IF    account ← accountRepository.findById(accountId)    IF account does not exist THEN        THROW AccountNotFoundException    END IF    IF account.status equals DELETED THEN        THROW AccountAlreadyDeletedException    END IF    previousStatus ← account.status    account.softDelete(adminId, reason, CURRENT_TIME)    BEGIN TRANSACTION        accountRepository.save(account)        sessionService.revokeAllSessions(accountId)    COMMIT TRANSACTION    RETURN DeleteAccountResult(        accountId,        previousStatus,        DELETED,        account.deletedAt,        "Account deleted successfully"    )END FUNCTION
00671: Operation 2 – Account.softDelete()
00672: FUNCTION softDelete(deletedBy, reason, currentTime)    IF current status equals DELETED THEN        THROW AccountAlreadyDeletedException    END IF    IF current status is not ACTIVE       AND current status is not INACTIVE       AND current status is not LOCKED THEN        THROW InvalidAccountStateException    END IF    IF reason is null OR reason is empty THEN        THROW DeletionReasonRequiredException    END IF    status ← DELETED    this.deletedBy ← deletedBy    this.deletedAt ← currentTime    this.deleteReason ← reason    this.updatedAt ← currentTimeEND FUNCTION

00679: 12.3. Service
00680: @Transactionalpublic DeleteAccountResult softDeleteAccount(        Long accountId,        Long adminId,        String reason) {    permissionService.requirePermission(adminId, "MANAGE_ACCOUNT");    if (accountId.equals(adminId)) {        throw new SelfDeletionNotAllowedException();    }    if (reason == null || reason.isBlank()) {        throw new DeletionReasonRequiredException();    }    Account account = accountRepository.findById(accountId)        .orElseThrow(AccountNotFoundException::new);    AccountStatus previousStatus = account.getStatus();    account.softDelete(adminId, reason, LocalDateTime.now());    accountRepository.save(account);    sessionService.revokeAllSessions(accountId);    return DeleteAccountResult.of(account, previousStatus);}
00681: 12.4. Entity
00682: public void softDelete(        Long deletedBy,        String reason,        LocalDateTime currentTime) {    if (status == AccountStatus.DELETED) {        throw new AccountAlreadyDeletedException();    }    if (status != AccountStatus.ACTIVE            && status != AccountStatus.INACTIVE            && status != AccountStatus.LOCKED) {        throw new InvalidAccountStateException(status);    }    if (reason == null || reason.isBlank()) {        throw new DeletionReasonRequiredException();    }    this.status = AccountStatus.DELETED;    this.deletedBy = deletedBy;    this.deletedAt = currentTime;    this.deleteReason = reason;    this.updatedAt = currentTime;}
